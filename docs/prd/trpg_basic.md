# 📜 상세 제품 요구사항 명세서 (PRD): 웹 기반 다중 에이전트 TRPG 플랫폼

# **📜 상세 제품 요구사항 명세서 (PRD): 웹 기반 다중 에이전트 TRPG 플랫폼**

## **1. 프로젝트 개요 및 비전 (Project Overview)**

- **목표:** 1~7인의 플레이어가 단일 웹 페이지에 동시 접속하여, 생성형 AI(Gemini API)가 구동하는 게임 마스터(GM) 및 NPC들과 실시간으로 상호작용하는 턴제 텍스트 TRPG 플랫폼 개발.
- **코어 밸류:** 발더스 게이트 3 스타일의 '개인화된 동적 선택지'를 통해 TRPG의 진입 장벽을 낮추면서도, '직접 입력'을 통한 무한한 자유도를 동시에 제공.
- **초기 테마:** 정통 하이 판타지 (마을 외곽 폐허의 고블린 소탕 및 유물 회수). 이후 미스터리, 호러, SF 등으로 확장 가능한 범용적 엔진 구축.

## **2. 핵심 게임플레이 시스템 (Core Gameplay Mechanics)**

### **하이브리드 행동 선언 시스템:**

- 플레이어는 게임 진입 전 간단한 성향 테스트(MBTI, 에니어그램, D&D 가치관)를 거쳐 자신의 페르소나를 확정.
- 플레이어 턴 도래 시, 시스템은 해당 캐릭터의 성향에 완벽히 부합하는 **3개의 맞춤형 행동 선택지**를 동적으로 생성하여 버튼으로 제공 (예: 질서 선 ENFJ 유저에게는 '외교/보호' 선택지, 혼돈 중립 ISTP 유저에게는 '기만/암살' 선택지 제공).
- 자유도 보존을 위해 **1개의 '직접 행동 선언(Text Input)'** 영역을 상시 노출.

### **판정 및 턴 제어 로직:**

- 별도의 주사위 엔진 대신, AI GM이 상황 맥락과 캐릭터 스탯을 바탕으로 [대성공/성공/부분 성공(대가성)/실패]를 객관적으로 판정.
- 서버 사이드에서 플레이어와 NPC 간의 순서를 강제하는 라운드 로빈(Round-Robin) 방식을 채택하며, AFK(자리 비움) 유저를 대비해 30초 등 제한 시간(Timeout) 만료 시 자동 방어 및 턴 넘김 처리.

## **3. AI 및 멀티 에이전트 아키텍처 (AI Architecture)**

### **역할별 에이전트 분리 원칙:**

- **System GM 에이전트:** 룰 통제, 물리 법칙 적용, 행동 판정 전담. 묘사(Narrative)와 시스템 상태 변화(HP, 어그로 등)를 엄격한 **JSON 포맷**으로만 반환.
- **NPC 에이전트:** 철저히 극 중 캐릭터로서만 발화하며, 게임 시스템을 통제할 권한 없음.

### **사전 생성형 NPC 파이프라인:**

- 시나리오 시작 시 백엔드에서 필요한 NPC 슬롯에 [역할 + MBTI + 에니어그램 + D&D 성향] 파라미터를 조합해 프로필(이름, 외형, 성격, 숨겨진 동기)을 사전 생성하고 DB에 적재하여 일관성 유지.

### **서사적 무결성 (미스터리 장르 대응):**

- 범인, 유물의 위치, 사건의 진상 등 '핵심 진실'은 슈뢰딩거 방식(실시간 변동)이 아닌, 세션 시작 시 **고정 데이터로 프롬프트에 하드코딩**하여 AI의 환각(Hallucination)과 설정 붕괴 차단.

## **4. 시스템 구조 및 데이터베이스 설계 (System & Database)**

### **실시간 통신:**

- WebSocket 또는 SSE(Server-Sent Events)를 통해 모든 유저의 화면에 턴 진행 상황과 AI의 텍스트 스트리밍을 지연 없이 동기화.

### **데이터베이스 스키마 핵심 엔티티:**

| **테이블명** | **핵심 역할** | **주요 컬럼** |
| --- | --- | --- |
| `Scenario` | 세계관 및 퀘스트 마스터 데이터 | 장르(Theme), 글로벌 GM 프롬프트, 클리어 조건 |
| `NPC_Persona` | 자동 생성된 NPC 메타데이터 | 역할, 페르소나 지시어, 숨겨진 동기(JSON) |
| `Game_Session` | 방(Room) 단위의 게임 상태 | 현재 턴 유저 ID, 타임아웃 시간, 세션 상태 |
| `Player_Character` | 유저 캐릭터 정보 | 직업, 성향 데이터, 인벤토리/스탯(JSON) |
| `Action_Log` | 휘발성 채팅 및 행동 로그 | 발화자 ID, 텍스트(행동 선언/판정 결과) |
| `Session_Memory` | 토큰 관리를 위한 요약 컨텍스트 | 요약된 스토리를 담은 영구 기억 RAG 텍스트 |

### **API 통신 페이로드 (JSON 기반):**

- 클라이언트 ➔ 서버: `[발화자 ID]`, `[행동 유형(선택지/직접입력)]`, `[상태 데이터]` 전송.
- 서버 ➔ Gemini API: 발화자를 명확히 태깅(`[Player_A]: ~한다`)하고, `Session_Memory`를 컨텍스트로 주입하여 호출.
- 서버 ➔ 클라이언트: AI가 반환한 JSON을 파싱하여, 클라이언트 UI에 묘사 출력 및 상태 게이지(HP 등) 업데이트 브로드캐스트.

## **5. 향후 개발 구현 마일스톤 (Next Development Milestones)**

### **Phase 1: 백엔드 코어 및 통신 뼈대 구축**

- Node.js/Python 등을 활용한 WebSocket/SSE 실시간 브로드캐스팅 서버 세팅.
- `Game_Session` 기반의 상태 머신(턴 큐 순서 강제, 타임아웃 타이머) 로직 구현.•

### **Phase 2: AI 에이전트 파이프라인 통합**

- Gemini API 연동 및 GM 에이전트의 JSON 강제 출력(Structured Output) 프롬프트 튜닝.
- [MBTI + 에니어그램 + 성향] 조합을 통한 NPC 프로필 자동 생성 스크립트 작성.
- 턴 도래 전, 플레이어 성향 기반 3가지 동적 선택지를 백그라운드에서 사전 생성하는 API 호출 로직 구현.

### **Phase 3: 메모리 최적화 및 DB 연동**

- 토큰 초과 방지를 위한 텍스트 요약(Summarization) 파이프라인 구축. N턴 주기로 `Action_Log`를 압축하여 `Session_Memory`에 업데이트.
- 제시된 6개 핵심 테이블 DB(PostgreSQL/MySQL 등) 스키마 생성.

### **Phase 4: 프론트엔드 UI/UX 개발**

- 접속, 캐릭터 성향 테스트, 대기실 UI 구현.
- 게임 진행 화면(텍스트 스트리밍 채팅창, 동적 선택지 렌더링, 캐릭터 상태창) 개발.
