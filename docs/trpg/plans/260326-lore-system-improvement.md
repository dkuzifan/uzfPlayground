## 1. 배경 및 문제 정의

### 1-1. 현재 Lore 시스템 구조

이 프로젝트의 Lore 시스템은 두 레이어로 구성된다.

**제작 레이어 (Scenario Creation)**
- Step F에서 `generate-lore` API가 시나리오 정보를 받아 Lore 항목 8~12개를 한 번에 생성
- 제작자가 개별 편집 후 저장
- DB 테이블: `World_Dictionary`

**런타임 레이어 (During Play)**
- `lore-engine.ts`가 플레이어 입력 텍스트에서 `trigger_keywords`를 스캔
- 매칭된 Lore 항목을 NPC 프롬프트에 동적으로 주입
- 토큰 방어(TOKEN_CAP_CHARS = 300자), 태그 군집화, Lore Queue로 관리

이 구조는 잘 설계되어 있으나, 한 가지 결정적인 빈틈이 있다.

### 1-2. 핵심 문제: "일회성 생성 이후 품질 검증이 없다"

`generate-lore`는 시나리오 정보를 기반으로 Lore를 처음 생성한다. 그러나 이후 **목표, 엔딩, NPC, 게임 룰이 추가되거나 수정**되어도 Lore를 재검토하는 기회가 없다.

시나리오는 선형적으로 제작되지 않는다. 실제 제작 흐름은 다음과 같다.

```
기본 정보 작성
    ↓
GM 프롬프트 초안 작성
    ↓ (이 시점에 generate-lore 실행)
Lore 생성
    ↓
목표 수정 ("사원에서 수정 구슬 회수" → "사원을 파괴하라"로 변경)
    ↓
엔딩 조건 추가 (비밀 엔딩: "반역자 정체 폭로")
    ↓
NPC 추가 (정보상 캐릭터 추가)
    ↓  ← 여기서 Lore 재검토가 없음
저장
```

결과적으로 Lore가 최종 시나리오와 어긋나는 상태로 저장될 수 있다.

## 2. TRPG 설계 원칙과의 관계

### 2-1. "세계를 준비하되 이야기는 열어두어라"

TRPG 설계에서 널리 받아들여지는 원칙이 있다.

> **"Prep the world, not the plot."**
> 이야기의 순서는 미리 짜지 마라. 하지만 세계는 준비해라.

좋은 시나리오는 *어떤 사건이 어떤 순서로 일어날 것인지*를 강제하지 않는다. 대신 **장소, 인물, 비밀, 갈등 구조**가 탄탄하게 준비되어 있으면, 플레이어가 어떤 방향으로 움직이더라도 GM이 일관된 세계로 반응할 수 있다.

Lore 시스템이 하는 일이 정확히 이것이다. Lore는 이야기를 강제하는 레일이 아니라, **AI GM이 어떤 상황에서도 일관된 세계를 표현할 수 있게 하는 재료**다.

### 2-2. AI GM과 인간 GM의 즉흥성 차이

인간 GM은 즉흥적으로 설정을 만들어도 뇌가 기억한다. 세션 내내 일관성을 유지할 수 있다.

AI GM은 다르다. 매 턴 컨텍스트(GM 프롬프트 + Lore + NPC 메모리 + 최근 로그)만으로 판단한다. Lore에 없는 내용을 즉흥으로 만들면, **컨텍스트 윈도우가 밀릴수록 그 설정을 잊고 모순된 응답을 낼 수 있다.**

```
플레이어 (턴 3): "북쪽 숲에 뭔가 있나요?"
AI (Lore 없음):  "오래된 나무꾼의 오두막이 보입니다."

플레이어 (턴 11): "그 오두막 주인은 누구였나요?"
AI (앞 내용 소실): "이 숲에는 특별한 구조물이 없습니다."
```

이 불일치가 TRPG의 몰입을 깨뜨린다. **준비된 Lore가 많을수록 AI는 더 일관되게 세계를 표현하고, 오히려 더 자연스러운 즉흥 반응이 가능해진다.**

### 2-3. Lore가 창의성을 해치지 않는 이유

Lore 구조화에 대한 우려가 있을 수 있다.

> "모든 걸 미리 정해두면 GM의 자유도가 줄어들지 않나?"

이 우려는 인간 GM에게는 타당하다. 그러나 이 시스템에서 즉흥성의 주체는 GM이 아니라 **플레이어**다.

- "설득 대신 협박을 시도한다"
- "문 대신 창문으로 진입한다"
- "편지에 숨겨진 잉크가 있는지 확인한다"

이런 예상 밖의 플레이어 행동이 TRPG의 핵심 재미다. Lore가 잘 구축된 세계일수록 AI는 이런 행동에 **설득력 있고 일관된 반응**을 돌려줄 수 있다. Lore는 제약이 아니라 즉흥의 **토대**다.

단, 한 가지 진짜 위험은 있다: **"Lore를 다 채워야 한다"는 압박이 진입장벽이 되는 것.** 이 기능은 반드시 제안/선택 방식이어야 하며, 통과 조건이 되어서는 안 된다.

## 3. 기능 정의: 시나리오 완성도 기반 Lore 보완 제안

### 3-1. 한 줄 정의

> AI가 완성된 시나리오(목표 + 엔딩 + NPC + 룰)를 분석하여, **이 게임이 실제로 진행될 때 플레이어가 마주칠 상황들을 예측**하고, 그 상황을 소화하기 위해 필요한데 Lore에 없는 항목을 제안한다.

### 3-2. 기존 generate-lore와의 차이

| 항목 | generate-lore (기존) | Lore 보완 제안 (신규) |
|------|------|------|
| 실행 시점 | Lore가 없는 초기 상태 | 이미 Lore가 일부 있는 상태 |
| 입력 | 시나리오 텍스트 | 시나리오 **전체** + **기존 Lore 목록** |
| AI 역할 | 처음부터 생성 | 플레이 예측 → 기존 대조 → 빈틈만 제안 |
| 중복 방지 | 없음 | 기존 Lore와 겹치면 제안 안 함 |
| reason 제공 | 없음 | "왜 이 Lore가 필요한가" 설명 포함 |
| 제작자 선택권 | 전체 수용 or 전체 거부 | per-card accept / dismiss |

### 3-3. AI가 수행하는 분석 로직 (프롬프트 설계 방향)

AI는 다음 3단계로 분석한다.

**Step 1: 플레이어 행동 시뮬레이션**
시나리오의 목표 유형, 엔딩 조건, NPC 구성을 읽고, 플레이어들이 이 시나리오에서 자주 취할 행동 유형을 예측한다.
- 목표가 `find` (정보 발견) → 플레이어가 NPC에게 정보를 캐는 행동이 많을 것
- 목표가 `eliminate` (제거) → 적대 세력의 배경, 약점, 근거지 관련 질문이 많을 것
- 비밀 목표가 있음 → 플레이어가 눈치채기 전 복선 Lore가 필요

**Step 2: 기존 Lore 커버리지 대조**
예측된 행동 패턴이 현재 Lore로 커버되는지 확인한다. 구체적으로:
- 예측된 질문 키워드가 `trigger_keywords`에 존재하는가?
- 엔딩 직결 요소(`importance_weight 8~10`)가 Lore에 등록되어 있는가?
- NPC별로 알고 있을 만한 비밀(`PERSONAL_LORE`)이 존재하는가?

**Step 3: 빈틈 항목 생성 + reason 작성**
커버되지 않는 항목에 대해 Lore를 생성하고, 각 항목마다 reason을 함께 작성한다. reason은 다음을 포함한다:
- 이 Lore가 어떤 상황에서 활성화될 것인지
- 없으면 어떤 문제가 생기는지

## 4. UI/UX 설계

### 4-1. 진입점

Step F (세계관 Lore) 하단에 새 버튼 추가.

```
[기존 Lore 목록]

────────────────────────────────────────
✨ AI 보완 분석   ← 새 버튼
  이 시나리오에서 누락된 Lore를 AI가 제안합니다.
────────────────────────────────────────
```

기존 "✨ AI 자동 생성" 버튼과 병존한다. 용도 구분:
- **AI 자동 생성**: Lore가 없는 초기 상태에서 전체 생성
- **AI 보완 분석**: Lore가 일부 있는 상태에서 빈틈만 제안

### 4-2. 제안 카드 UI

분석 완료 시 제안 카드 목록이 표시된다.

```
┌─────────────────────────────────────────────────┐
│ AI 보완 제안 (3개)                               │
│                                                 │
│ ┌─────────────────────────────────────────────┐ │
│ │ WORLD_LORE · 역사                  [채용] [✕] │ │
│ │                                             │ │
│ │ "성벽 동쪽 탑은 15년 전 역병으로 봉쇄된     │ │
│ │  이후 아무도 출입하지 않는다."              │ │
│ │                                             │ │
│ │ trigger: ["동쪽 탑", "역병", "봉쇄"]        │ │
│ │                                             │ │
│ │ 💡 비밀 목표 '탑 안의 기록 발견'과 직결.    │ │
│ │    플레이어가 탑에 대해 물었을 때 NPC가     │ │
│ │    답할 근거가 현재 없습니다.               │ │
│ └─────────────────────────────────────────────┘ │
│                                                 │
│ ┌─────────────────────────────────────────────┐ │
│ │ PERSONAL_LORE · 비밀               [채용] [✕] │ │
│ │  ...                                        │ │
│ └─────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────┘
```

**동작 규칙**:
- [채용]: 해당 Lore를 현재 목록 맨 아래에 추가
- [✕]: 해당 제안만 목록에서 제거 (다른 제안에 영향 없음)
- 전체 닫기 버튼 없음 → 하나씩 처리 유도
- 이미 채용한 항목은 "채용됨" 배지로 표시

## 5. API 설계

### 5-1. 엔드포인트

`POST /api/trpg/scenarios/analyze-lore-gaps`

### 5-2. Request Body

```typescript
{
  title: string;
  theme: string;
  description?: string;
  gm_system_prompt?: string;
  objectives?: ScenarioObjectives;
  endings?: ScenarioEndings;
  npc_names?: string[];          // NPC 이름 목록 (공개 정보만)
  stat_schema?: StatSchemaEntry[];
  existing_lore: {               // 현재 등록된 Lore 요약
    category: string;
    lore_text: string;
    trigger_keywords: string[];
  }[];
}
```

### 5-3. Response Body

```typescript
{
  suggestions: Array<{
    domain: "WORLD_LORE" | "PERSONAL_LORE";
    category: string;
    lore_text: string;
    trigger_keywords: string[];
    cluster_tags: string[];
    importance_weight: number;
    required_access_level: number;
    reason: string;              // 왜 이 Lore가 필요한가
  }>
}
```

### 5-4. 프롬프트 핵심 지시사항

- 기존 Lore와 중복되는 항목은 제안하지 말 것
- 제안 수는 3~6개로 제한 (너무 많으면 선택 피로도 증가)
- reason은 1~2문장, 구체적인 게임 상황으로 설명할 것
- `importance_weight`가 7 이상인 항목 (엔딩 직결)을 우선 제안할 것

## 7. 설계 원칙 요약

1. **선택적 깊이**: Lore가 0개여도 게임은 작동한다. 이 기능은 품질을 높이는 도구이지 통과 조건이 아니다.
2. **제안이지 강제가 아님**: per-card accept/dismiss. 제작자가 최종 결정권을 가진다.
3. **중복 없음**: 이미 있는 Lore는 다시 제안하지 않는다.
4. **reason 필수**: "왜 이 Lore가 필요한가"를 항상 함께 표시하여, 제작자가 맥락을 이해하고 채용 여부를 판단할 수 있게 한다.
5. **간결함**: 제안 수 3~6개로 제한. Lore 시스템의 목적은 완전한 사전 문서화가 아니라 AI의 일관성 향상이다.
