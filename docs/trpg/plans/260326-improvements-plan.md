## 카테고리별 작업 목록

---

### 🔴 A. 핵심 기능 공백

> 게임이 설계 의도대로 완전히 동작하지 않는 부분.

---

#### A-1. NPC 설계 시나리오 통합 ⭐ 최우선

**현황**: NPC_Persona 생성이 시나리오 제작 플로우(Step A~G) 밖에 존재.
제작자는 시나리오를 완성한 뒤 완전히 별도로 NPC를 만들어야 함.

**문제**:
- Lore 보완 분석(`analyze-lore-gaps`)이 `npc_names`를 빈 배열로 보낼 수밖에 없음 → PERSONAL_LORE 제안 품질 저하
- NPC 없이는 게임 자체가 의미 없음 (이 게임의 핵심이 NPC와의 상호작용)
- 목표/엔딩/Lore가 NPC와 연결되어 있는데 제작 시 이 연결을 검토할 수 없음

**작업 범위**:
- 시나리오 제작에 **Step H: NPC 설계** 추가
- per-NPC: 이름, 역할, 외형, 성격, 숨겨진 동기, resistance_stats, 언어 프로필
- AI 자동 생성 버튼 (`generate-npc` API 또는 기존 `/api/trpg/npc/generate` 활용)
- 완성된 NPC 목록을 `analyze-lore-gaps`에 전달

**참조**: `docs/plans/260326-lore-system-improvement.md` §3-2

---

#### A-2. 시나리오 직접 수정

**현황**: 기존 시나리오는 "복사본 만들기"만 가능. 수정 API(PUT) 없음.

**문제**:
- 시나리오를 조금 수정하려 해도 전체를 새로 제작해야 함
- 복사본이 쌓이면 관리 혼란

**작업 범위**:
- `PUT /api/trpg/scenarios/[scenarioId]` 구현
- 시나리오 목록에서 "수정" 버튼 → ScenarioCreateStep을 수정 모드로 진입
- 수정 모드: 기존 데이터 로드 후 저장 시 INSERT 대신 UPDATE

---

#### A-3. generate-rules info_rules 타입 불일치 (minor bug)

**현황**: `generate-rules` API 응답이 `use_private_info: boolean` (단일값)을 반환하는데, 실제 `InfoRules` 타입은 `{ private_items: boolean, private_lore: boolean }` (분리).
현재 `acceptInfoRuleSuggestion()`이 둘 다 동일값으로 설정하는 임시 처리 중.

**작업 범위**:
- `generate-rules` 프롬프트와 응답 스키마를 `private_items` / `private_lore` 분리 반환으로 수정
- AI가 "아이템은 비공개, Lore는 공개"처럼 세분화된 제안 가능하도록

---

### 🟡 B. 게임플레이 경험 개선

> 게임은 돌아가지만, TRPG로서의 재미가 아직 충분하지 않은 부분.

---

#### B-1. GM 전용 인게임 패널

**현황**: 방장(호스트)이 게임 중 일반 플레이어와 동일한 화면만 봄. GM 고유 도구 없음.

**TRPG 관점**: 인간 GM은 게임 중 언제든 세계를 조정할 수 있어야 함. AI GM이 주도하는 이 시스템에서도 호스트가 개입할 수 있는 창구가 필요.

**필요 기능**:
- NPC 심리변수 수동 조정 (예: 특정 NPC 신뢰도 강제 변경)
- 내러티브 직접 주입 (GM이 서술을 직접 삽입)
- Doom Clock 수동 조작
- 턴 강제 진행 (비활성 플레이어 건너뜀)
- 인게임 Lore 즉석 추가 (자연어 → World_Dictionary 포맷 자동 변환)

**작업 범위**:
- 게임 화면 우측 사이드바에 GM 전용 슬라이드 패널 (호스트만 표시)
- 각 기능별 API 엔드포인트 (일부는 기존 Supabase 직접 업데이트)

---

#### B-2. Lore 발견 드라마 UI

**현황**: `lore_discovery` 타입 로그가 채팅 스트림에 일반 메시지처럼 흘러감.

**TRPG 관점**: "숨겨진 것을 찾아냈다"는 순간은 게임의 가장 강한 감정적 피크 중 하나. 시각적으로 강조되지 않으면 발견의 성취감이 사라짐.

**작업 범위**:
- ChatLog에서 `lore_discovery` 타입을 별도 카드 형식으로 렌더링
- 📜 아이콘, 강조 색상, "새로운 단서 발견" 헤더
- 사이드바에 "발견한 단서 목록" 소형 패널 추가 (게임 중 누적)

---

#### B-3. 멀티플레이 경험 개선

**현황**: 멀티플레이가 사실상 "순서를 기다리는 솔로 플레이 N개".

**TRPG 관점**: 같은 테이블에 앉은 느낌을 만드는 것이 멀티 TRPG의 핵심.

**필요 기능**:

| 기능 | 설명 | 구현 난이도 |
|------|------|------------|
| 반응 버튼 | 다른 플레이어 턴 중 😨 🎉 ⚠️ 같은 감정 반응 전송 | 낮음 |
| 보조 행동 | 현재 턴 플레이어에게 "지원 선언" → 판정 보너스 | 중간 |
| 주사위 롤 공유 | 다른 플레이어가 주사위 굴릴 때 내 화면에도 애니메이션 표시 | 중간 |

---

#### B-4. NPC 자발적 행동 (이벤트 트리거)

**현황**: NPC는 항상 플레이어 행동에만 반응. 세계가 플레이어 없이 움직이지 않음.

**TRPG 관점**: 살아있는 세계는 플레이어가 아무것도 안 해도 변화가 일어나야 한다.

**설계 방향**:
- `Game_Session.npc_dynamic_states`의 변수 임계값에 따라 자동 트리거
  - `fear_survival ≥ 80` → NPC 도주/항복 선언
  - `affinity ≥ 90` → NPC가 먼저 비밀 정보 공유
  - N턴 비상호작용 → NPC가 먼저 말을 걸어옴
- 트리거 조건은 시나리오 설계 시 제작자가 커스텀 정의 가능

---

#### B-5. 세션 리캡

**현황**: 게임 종료 → EndingScreen → 로비. 완료 세션을 다시 볼 수 없음.

**작업 범위**:
- 완료된 세션의 주요 Action_Log 요약 페이지
- 발견한 Lore, 획득 아이템, 주요 결정 순간, 최종 엔딩 표시
- `/trpg/sessions/[sessionId]/recap` 라우트

---

### 🟢 C. 플랫폼 완성도 (선택적)

> 개인 포털 프로젝트 범위를 넘어설 때 고려.

| 항목 | 설명 | 우선순위 |
|------|------|----------|
| 시나리오 공개/갤러리 | 다른 사용자의 공개 시나리오 탐색 및 플레이 | 낮음 |
| 플레이어 계정 영속성 | 현재 게스트(localStorage). 계정 시스템 연동 | 낮음 |
| 모바일 반응형 | 게임 화면 데스크탑 고정 레이아웃 → 반응형 | 낮음 |

---

### ⚠️ D. 기술 부채 / 보류 항목

| 항목 | 현황 | 결정 |
|------|------|------|
| 클라이언트 주사위 서버 검증 | rolled 값 서버 미검증 → 조작 가능 | 개인 프로젝트 규모상 보류 |
| GM 프롬프트 복잡도 | 선택지+서술 동시 생성 → 품질 분산 리스크 | 실플레이 후 모니터링 |
| 선택지 에이전트 분리 | GM이 선택지 실패 시 fallback API 호출 | 현행 유지, 안정적이면 OK |

## 권장 진행 순서

```
┌─────────────────────────────────────────────────────────────────┐
│  Phase 1 (기능 완성)                                              │
│                                                                 │
│  A-1 NPC 통합  →  A-2 시나리오 수정  →  A-3 info_rules 수정     │
│  (3~4주)           (1주)                 (0.5일)                │
└────────────────────────────┬────────────────────────────────────┘
                             │
┌────────────────────────────▼────────────────────────────────────┐
│  Phase 2 (게임성 개선)                                            │
│                                                                 │
│  B-2 Lore 발견 UI  →  B-1 GM 패널  →  B-3 멀티플레이           │
│  (1~2일)               (1주)           (1~2주)                  │
└────────────────────────────┬────────────────────────────────────┘
                             │
┌────────────────────────────▼────────────────────────────────────┐
│  Phase 3 (완성도)                                                 │
│                                                                 │
│  B-4 NPC 이벤트  →  B-5 세션 리캡  →  C. 플랫폼 확장           │
└─────────────────────────────────────────────────────────────────┘
```

**A-1이 최우선인 이유**:
- 게임의 핵심 재미(NPC 상호작용)를 제작자가 설계할 수 없는 상태
- Lore 보완 분석 품질이 NPC 유무에 직접 의존
- A-1 완료 후 B-1(GM 패널), B-4(NPC 이벤트)가 자연스럽게 연결됨
