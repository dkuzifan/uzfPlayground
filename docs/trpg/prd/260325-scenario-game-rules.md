# 시나리오 게임 룰 레이어 기획

> **작성일:** 2026-03-25
> **관련 문서:** `docs/prd/trpg_basic_v2.md`, `docs/prd/260325-d20-decision-improvement.md`

---

## 1. 왜 이게 필요한가

### 지금 구조의 한계

현재 시나리오를 만들 때 정의할 수 있는 것:

> **세계관 설정 + NPC + 목표 + 엔딩**

그런데 실제 TRPG를 생각해보면, 이것만으로는 부족하다. 게임을 실제로 "어떻게 플레이하는가"에 대한 규칙이 없기 때문이다.

예를 들어:
- 추리 미스터리 시나리오를 만들었는데, 모든 단서가 모든 플레이어에게 공개된다면? → 플레이어 A가 발견한 중요한 증거를 B가 바로 알게 되어 긴장감이 사라진다.
- 정치 음모 시나리오인데, 플레이어들이 서로 배신하거나 협력할 수 없다면? → 시나리오의 핵심 재미가 구현되지 않는다.
- 생존 호러 시나리오인데, 정신력(공포)이라는 개념이 없다면? → 호러의 느낌이 나지 않는다.

지금은 이 모든 것이 코드에 고정(하드코딩)되어 있다. 어떤 시나리오를 만들어도 게임 규칙은 똑같이 돌아간다.

### 실제 TRPG가 하는 방식

유명한 TRPG 시스템들은 "룰북"과 "시나리오 모듈"을 분리한다.

- **D&D**: 룰북(스탯/판정/전투 규칙) + 모험 모듈(세계관/NPC/스토리)
- **Call of Cthulhu**: 룰북(정신력/스킬 시스템) + 시나리오(사건/단서/진실)
- **Blades in the Dark**: 기본 규칙(포지션/효과) + 각 캠페인의 특수 규칙

이 프로젝트에서도 같은 구분이 필요하다. **시나리오를 만들 때, 그 시나리오에 맞는 게임 규칙도 함께 설계하는 것.**

## 2. 게임 룰 레이어란 무엇인가

시나리오 제작 시 정의하는 "이 게임에서 어떤 규칙이 적용되는가"에 대한 설정 묶음이다.

크게 4가지 영역으로 나뉜다.

---

### 영역 A: 정보 구조 규칙 (Information Rules)

**"플레이어들이 어떤 정보를 서로 공유하는가, 숨기는가?"**

현재는 모든 정보가 전체 공개다. 이걸 시나리오별로 설정할 수 있게 한다.

#### 설정 항목

**비공개 정보 사용 여부**
- OFF (기본): 모든 행동, 아이템 획득, 대화가 전체 공개
- ON: 특정 정보는 해당 플레이어에게만 표시

**비공개 대상 종류** (ON일 때)
- 아이템/단서 획득: "당신은 낡은 편지를 주웠다" → 이것을 공개할지 숨길지
- 특정 NPC와의 비밀 대화: 다른 플레이어가 자리를 비운 상황 설정 가능
- 특정 Lore 발견: 플레이어가 혼자 발견한 세계관 지식

**정보 공개 조건** (선택 사항)
- 플레이어가 직접 "공유"를 선택할 때 공개
- 특정 장소(씬)에 도달하면 자동 공개
- 특정 목표 달성 후 자동 공개

#### 실제 게임에서 어떻게 느껴지나

> 추리 미스터리 시나리오에서, 플레이어 A가 서재에서 혈흔 묻은 편지를 발견했다. 이 정보는 A에게만 표시된다. B와 C는 A가 서재에서 뭔가를 찾았다는 사실만 알고, 내용은 모른다. A는 이 정보를 언제 공유할지 전략적으로 결정할 수 있다.

---

### 영역 B: 특수 자원 규칙 (Resource Rules)

**"이 시나리오에서만 존재하는 특별한 수치나 자원이 있는가?"**

이미 구현된 유연한 스탯 시스템(StatSchemaEntry)의 확장이다. 하지만 그것이 "어떻게 변동되는가"에 대한 규칙도 함께 정의한다.

#### 설정 항목

**특수 자원 목록**
각 자원에 대해:
- 이름과 아이콘 (예: 정신력 🧠, 평판 ⭐, 증거 🔍, 피로도 😓)
- 초기값과 최대/최소값
- 증감 조건 설명 (예: "공포스러운 장면을 목격하면 정신력 감소")

**자원 고갈 시 효과**
- 정신력 0 → 행동 제한 또는 시나리오 실패 조건
- 평판 최대 → 특수 엔딩 활성화

#### 실제 게임에서 어떻게 느껴지나

> 호러 시나리오에서 플레이어는 정신력(Sanity) 수치를 가진다. NPC가 끔찍한 진실을 말할 때, GM이 정신력을 깎는다. 정신력이 0이 되면 캐릭터가 공황 상태에 빠져 행동 선택지가 바뀐다. 이것이 시나리오 설정에서 정의되어 있기 때문에, GM AI가 자동으로 이 규칙을 인식하고 적용한다.

---

### 영역 C: 행동 규칙 (Action Rules)

**"이 시나리오에서 어떤 행동이 허용되고, 어떤 행동에 특별한 효과가 있는가?"**

#### 설정 항목

**행동 카테고리 제한**
- 특정 카테고리 비활성화 가능 (예: 평화 시나리오에서 attack 금지)
- 특정 행동에 추가 조건 부여 (예: "persuade는 평판 10 이상일 때만 가능")

**특수 행동 효과**
- 특정 아이템 사용 시 행동 선택지 추가
- 특정 장소에서만 가능한 행동 정의

> 이 영역은 복잡도가 높아 구현 우선순위가 낮다. 기획 단계에서만 정의하고, 나중에 점진적으로 구현한다.

---

### 영역 D: 관계/사회 규칙 (Relationship Rules)

**"플레이어들 사이, 혹은 플레이어와 NPC 사이의 관계가 게임에 영향을 주는가?"**

#### 설정 항목

**플레이어 간 배신/협력**
- 협력 전용: 플레이어들은 항상 같은 팀 (기본값)
- 경쟁/배신 허용: 플레이어가 다른 플레이어를 방해하거나 NPC에게 밀고 가능

**파벌 시스템**
- 플레이어들이 서로 다른 세력에 소속될 수 있음
- 파벌 간 이해관계 충돌이 시나리오의 핵심인 경우

> 이 영역도 복잡도가 높아 나중에 구현한다.

## 3. AI 도움의 역할

시나리오 제작자가 직접 모든 룰을 설계하는 것은 부담이 크다. AI가 세계관과 테마를 분석해서 적합한 게임 룰을 제안하는 방식으로 보조한다.

### AI가 제안하는 방식

**입력**: 시나리오 제목, 테마, 시스템 프롬프트, 목표 설명
**출력**: 적합한 게임 룰 묶음 제안 + 각 룰에 대한 설명

#### 예시 — 추리 미스터리

> "밀실 살인 사건을 조사하는 탐정들" 시나리오를 분석했습니다.
>
> **추천 규칙:**
> - ✅ 비공개 정보 ON — 단서와 증거는 발견한 플레이어에게만 공개
> - ✅ 증거(Evidence) 자원 추가 — 조사 성공 시 획득, 최종 고발에 필요
> - ✅ 정보 공유 행동 추가 — 플레이어가 직접 단서를 다른 플레이어와 공유 선택 가능
> - ✅ attack 카테고리 비활성화 — 수사 시나리오에 물리 전투 부적절

#### 예시 — 생존 호러

> "폐쇄된 우주 정거장에서 탈출" 시나리오를 분석했습니다.
>
> **추천 규칙:**
> - ✅ 비공개 정보 ON — 공포 반응, 개인 트라우마는 비공개
> - ✅ 정신력(Sanity) 자원 추가 — 호러 이벤트 목격 시 감소, 0이 되면 패닉 상태
> - ✅ 피로도(Fatigue) 자원 추가 — 시간이 지날수록 증가, 판정에 페널티
> - ✅ Desperate 포지션 빈도 증가 — 전반적으로 불리한 상황 유지

제작자는 각 제안을 수락/거부/수정하면서 자신만의 룰을 완성한다.

## 4. 시나리오 제작 흐름 변화

### 현재 흐름 (6단계)

```
A. 기본 설정 (제목/테마)
B. 목표 설정
C. 엔딩 설정
D. 캐릭터 설정 (직업/스탯)
E. 커스텀 직업
F. 세계관 Lore
```

### 변경 후 흐름 (7단계)

```
A. 기본 설정 (제목/테마)
B. 목표 설정
C. 엔딩 설정
D. 캐릭터 설정 (직업/스탯)
E. 커스텀 직업
F. 세계관 Lore
G. 게임 룰 설정 ← 신규
```

Step G에서:
1. AI가 현재까지 입력된 시나리오 내용을 분석해서 룰을 자동 제안
2. 제작자가 수락/거부/수정
3. 직접 추가도 가능

## 5. 구현 계획 (단계별)

### Phase 1 — 정보 구조 규칙 (가장 즉각적인 효과)

**DB 변경:**
- `Action_Log` 테이블에 `is_private boolean` 컬럼 추가
- `Scenario` 테이블의 `character_config`에 `info_rules` 필드 추가

**서버 변경:**
- resolve route: `is_private` 여부를 시나리오 설정 기반으로 결정 후 INSERT
- 게임 페이지 세션 조회 API: `is_private = true`인 로그는 `speaker_id`가 본인인 경우에만 반환

**클라이언트 변경:**
- ChatLog: 비공개 로그를 본인에게는 🔒 아이콘과 함께 표시, 타인에게는 숨김
- 비공개 정보를 공유하는 버튼 추가 (선택적)

**시나리오 제작:**
- Step G UI에 "비공개 정보 사용" 토글 추가

---

### Phase 2 — 특수 자원 규칙 (이미 stat_schema 기반 있음)

**이미 된 것:**
- 유연한 스탯 스키마(StatSchemaEntry)로 어떤 자원이든 표시 가능

**추가로 필요한 것:**
- 자원 변동 규칙 정의 — "어떤 이벤트에서 얼마나 변하는가"
- GM AI에게 자원 변동 규칙 주입 — GM이 자동으로 자원을 조정
- resolve route에서 자원 변동을 state_changes에 반영

---

### Phase 3 — AI 룰 제안 (generate-rules API)

**새 API:** `POST /api/trpg/scenarios/generate-rules`
- 입력: 시나리오 기본 정보 전체
- 출력: 추천 규칙 묶음 (JSON)

**시나리오 제작 Step G UI:**
- "AI 룰 제안 받기" 버튼
- 제안된 룰 카드 목록 (각각 수락/거부 가능)
- 직접 추가 폼

---

### Phase 4 — 행동/관계 규칙 (추후)

복잡도가 높아 Phase 1~3 완료 후 필요에 따라 추가.

---

## 6. 각 단계의 게임 경험 변화

### Phase 1 이후

추리 미스터리를 만들면 진짜 추리 미스터리가 된다. 플레이어마다 다른 정보를 가지고 협력하거나 경쟁하는 구조가 생긴다.

### Phase 2 이후

시나리오 테마가 실제 게임 수치로 느껴진다. 호러 시나리오에서는 공포가 정신력 수치로 가시화되고, 생존 시나리오에서는 자원 관리가 핵심 전략이 된다.

### Phase 3 이후

시나리오 제작자가 게임 디자이너가 된다. AI가 도와주기 때문에 전문 지식 없이도 세계관에 맞는 게임 룰을 설계할 수 있다. 추리 미스터리, 호러, 정치 음모, 액션 어드벤처가 각자 다른 게임처럼 느껴진다.

## 7. 현재 시스템과의 연결

이미 구현된 것들이 이 기획과 자연스럽게 연결된다:

| 기존 구현 | 이 기획에서의 역할 |
|----------|------------------|
| StatSchemaEntry (유연한 스탯) | Phase 2 특수 자원의 표시 레이어 |
| NPC 심리 상태 17변수 | 관계 규칙의 데이터 기반 |
| Position 시스템 | 자원 상태에 따라 Position이 변동되는 확장 가능 |
| World_Dictionary Lore | 비공개 Lore 발견 → Phase 1에서 is_private 적용 대상 |
| Lore AI 자동 제안 | generate-rules AI와 동일한 패턴으로 구현 가능 |
