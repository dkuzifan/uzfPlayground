## Context

야구 시뮬레이터 수비 엔진 시리즈 #7. 현재 모든 수비수는 `FIELDER_DEFAULT_POS`에 고정된 기본 좌표에 서 있다. 타자의 타구 경향과 무관하게 항상 같은 위치에서 수비하는 구조이므로, 실제 야구에서 핵심 전략인 "수비 시프트"가 전혀 반영되지 않는다.

수비 시프트는 타자의 타구 방향 편향(주로 풀히팅 경향)을 사전에 예측하고 수비수를 재배치함으로써 아웃 확률을 높이는 전략이다. MLB에서는 좌타 풀히터에 대한 강한 내야 시프트가 일반화된 뒤 2023년부터 규제(Shift Ban)될 정도로 효과가 검증된 전략이다.

현재 수비 엔진에서 `Player.defence_pos` 필드가 이미 optional로 선언되어 있으며, `findResponsibleFielder`와 `calcCatchProbability`가 이 값을 `FIELDER_DEFAULT_POS`보다 우선해서 읽는 구조이므로, 타석 단위로 `defence_pos`를 재설정하는 것만으로 시프트를 반영할 수 있다.

**설계 방향 (이전 논의에서 결정):**
- **Option 1 (자동 시프트)**: 타자의 `bats` + `power` 스탯으로 `pull_tendency`를 파생해 시프트 발동 및 오프셋 결정. 별도 스탯 추가 없음.
- **Option 3 (타석 단위 적용)**: `runAtBat` 진입 전 수비 라인업에 시프트 좌표를 임시 오버라이드. 원본 Player 데이터 불변.
- **Option 2 (프리셋)**: 감독 모드 구현 시 별도 기획.

## Goals / Non-Goals

**Goals (목표):**
- `bats` + `power`로 파생한 `pull_tendency`를 기반으로 내야 + 외야 시프트를 자동 결정한다.
- 스위치히터(`bats: 'S'`)는 상대 투수의 `throws`로 실제 타석 방향을 결정해 시프트를 적용한다 (우투수 → 좌타석, 좌투수 → 우타석).
- 시프트 적용은 타석 단위 임시 오버라이드로 원본 Player 데이터를 변경하지 않는다.
- 시프트 결과(어떤 타자에게 어느 방향으로 시프트했는지)를 GameEvent로 기록한다.

**Non-Goals (비목표):**
- 감독이 수동으로 시프트를 지시하는 UI/인터페이스 (감독 모드 피처에서 처리).
- 시프트 통계 누적 (시즌 스탯 피처에서 처리).
- MLB Shift Ban 룰 적용 (시뮬레이터 특성상 적용 안 함).

## Success Definition

- 좌타 고파워 타자에게 내야수 + 외야수가 1루 방향으로 이동하고, 포구 확률과 송구 거리 계산이 변경된 좌표를 반영한다.
- 우타 고파워 타자에게는 반대 방향(3루 방향)으로 시프트가 발동된다.
- 스위치히터는 해당 타석의 실제 타석 방향(투수 `throws` 기준)으로 시프트가 적용된다.
- pull_tendency는 발동 임계치 없이 모든 타자에게 연속값으로 적용된다 — 수치가 낮으면 이동 폭이 작을 뿐, 항상 최적 위치로 배치된다.
- 좌우(x) 이동은 타구 방향 편향에 따르고, 깊이(y) 이동은 타자의 파워에 따라 결정된다.
- 타석마다 `shift` GameEvent가 로깅된다.
- 기존 시프트 없는 타석과 결과 형태가 동일하게 유지된다 (회귀 없음).

## Requirements

**Must-have (필수):**

**R1. pull_tendency 파생 (연속값)**
- [ ] `bats` + `power`로 pull_tendency를 계산한다 (0.0~1.0 범위, 연속값).
- [ ] 파생 공식: `base(bats) + power_bonus`
  - base: 우타('R') → 0.45, 좌타('L') → 0.45 (방향은 별도, 크기는 동일)
  - power_bonus: `(power - 50) / 100 * 0.5` (power 50 → +0.0, power 100 → +0.25)
  - 결과: clamp(0.0, 1.0)
- [ ] 스위치히터: 상대 투수 `throws`로 실효 타석 방향 결정 후 동일 공식 적용 (우투수 → effectiveBats='L', 좌투수 → effectiveBats='R')
- [ ] pull_tendency는 이동 폭의 스케일 계수로만 사용 — 발동 임계치 없음, 모든 타자에게 항상 적용

**R2. 시프트 오프셋 결정 (연속 스케일)**
- [ ] 좌우 오프셋(dx)과 깊이 오프셋(dy)을 pull_tendency로 스케일링한 값으로 결정
- [ ] 모든 수비수에게 **(dx, dy) 쌍을 동시에** 계산하여 2D 이동 위치를 결정한다.
  - **dx (좌우)**: effectiveBats 방향 × pull_tendency 스케일. 좌타(+x, 1루 방향), 우타(-x, 3루 방향). pull_tendency가 높을수록 이동 폭 확대.
  - **dy (깊이)**: power 기반. 고파워 → 외야 뒤로, 저파워 → 내야 앞으로.
  - dx와 dy는 독립적 선택이 아니라 타석마다 동시에 결정되는 벡터.
- [ ] 1B: dx = 0 (앵커), dy만 적용
- [ ] C, P: 시프트 비대상 (dx=0, dy=0)

**R3. 타석 단위 임시 적용**
- [ ] `applyShift(lineup, batter, pitcher)`: 원본 Player 배열을 복사 후 `defence_pos`만 오버라이드한 새 배열 반환 (원본 불변)
- [ ] 시프트 비대상 수비수(C, P)는 `defence_pos` 미설정 상태 그대로 복사 (FIELDER_DEFAULT_POS 폴백 유지)
- [ ] `runAtBat` 진입 전에 호출되어 `defenceLineup` 인수로 전달된다

**R4. 이벤트 기록**
- [ ] 타석마다 `shift` GameEvent emit: `{ direction: 'left'|'right'|'none', pull_tendency: number, shifted_positions: Position[] }`
- [ ] pull_tendency가 0에 가까워 실질적 이동이 없어도 이벤트는 항상 emit (추적 가능하도록)

**Nice-to-have (선택):**
- [ ] 시프트 발동 여부에 따른 `at_bat_result` 분포 변화 확인용 디버그 플래그
