## Context

### 현재 문제

**문제 1 — 플라이아웃 시 주자들이 이동하지 않는다**

`advanceRunners('out', ...)` → `fixedAdvance('out')` → 모든 주자 제자리.
실제 야구에서는 플라이볼이 포구되는 순간부터 주자들이 **태그업(귀루 후 진루)**을 시도할 수 있다.

```
현재: 외야 플라이아웃 + 3루 주자 → 3루 주자 그대로 (버그)
기대: 3루 주자 → 홈 도전 판단 → 성공 시 득점
```

**문제 2 — `in_air` BallState가 타입만 예약되어 있고 구현이 없다**

#3-b에서 `BallState.in_air` 타입을 예약했지만 실제 동작은 없다.
태그업에서는 **공이 공중에 있는 동안(`t_remaining`)** 주자 상태를 정확히 표현해야 한다.

**문제 3 — 라인 드라이브 병살이 없다**

라인 드라이브가 직접 포구되면 주자가 귀루하지 못해 아웃될 수 있다.
현재 이 상황이 전혀 처리되지 않는다.

**문제 4 — 희생플라이 통계가 없다**

외야 플라이아웃 + 태그업 득점 = 희생플라이(SAC F). 현재 스탯 기록이 없다.

## Goals / Non-Goals

### Goals (Must-have)

**G1. `in_air` BallState 구현**

공이 공중에 있는 동안의 상태. 포구 여부가 확정되기 전이므로 주자는 귀루 대기.

```typescript
{ phase: 'in_air'; t_remaining: number; fielder_pos: Vec2; fielder: Player }
// t_remaining: 공 비행 시간 (= hp.t_ball_travel)
// 포구 확정 → 즉시 held 전환 → 태그업 시작
```

**G2. 외야 플라이아웃 태그업 판단 — `resolveOutfieldFlyOut`**

`at_bat_result='out'`, `is_infield=false` 조건에서 태그업 로직 발동.

태그업 판단 로직 (`decideChallengeAdvance` 재사용):
```
ball_state = { phase: 'held', fielder: 외야수, fielder_pos: 포구 위치 }
              (포구 완료 = 외야수가 공을 쥐고 있음)

runner_dist = dist(현재 베이스 → 다음 베이스) - return_lead
              (귀루 시간 = return_lead / runner_speed 만큼 이미 지연)

decideChallengeAdvance(runner, runner_dist, ball_state, target, lineup)
```

주자 처리 순서: 3루 → 2루 → 1루 (앞 주자 우선, 충돌 방지).

**G3. `return_lead` — 귀루 지연 계산**

공이 날아가는 동안 주자는 이미 리드를 취하고 있음. 포구 후 귀루 시간만큼 태그업 출발이 늦어짐.

```
lead_dist    = calcPitchLead(runner)     // 현재 pitch lead 거리 (≈ 3~5m)
return_time  = lead_dist / runner_speed  // 귀루에 걸리는 시간
runner_dist  = dist(base → next_base)   // 태그업 진루 거리
// decideChallengeAdvance에서 t_runner = (runner_dist + lead_dist) / runner_speed
// 즉 귀루 시간이 t_runner에 더해져 자연스럽게 불리하게 작용
```

**G4. 라인 드라이브 병살 — `resolveLDDoublePlay`**

공 비행 시간이 짧은 경우(`t_ball_travel < LINE_DRIVE_THRESHOLD`) 포구 후 주자 귀루 아웃 가능.

```
귀루 시간 = lead_dist / runner_speed
포구 후 수비수 → 루 태그/송구 시간

귀루 시간 > 송구 시간 → 귀루 아웃 (라인 드라이브 병살)
귀루 시간 ≤ 송구 시간 → 귀루 성공 (태그업 대기)
```

발동 조건: `t_ball_travel < 0.8s` (라인 드라이브 기준값, 밸런싱 대상).

**G5. `catch_setup_time` 구현**

포구 난이도에 따른 송구 준비 시간. 태그업에서 `reaction_delay`로 사용.

```
일반 포구:       0.2s
어려운 포구:     0.4s  (diving catch 등, catch_probability < 0.5일 때)
```

`HitResultDetail.catch_setup_time`에 값 할당 (현재 예약만 된 상태).

**G6. 희생플라이 통계**

외야 플라이아웃 + 태그업 득점 시 타자에게 SAC F 기록.
기존 `GameStats`에 `sac_fly: number` 추가.

**G7. 2아웃 득점 타이밍 판정**

포구가 2번째 아웃일 때, 태그업 도중 3번째 아웃이 발생하면 홈인 타이밍 비교:
- `score_time < out_time` → 득점 유효
- `score_time >= out_time` → 득점 무효

### Non-Goals

| 항목 | 이유 |
|------|------|
| 내야 팝플라이 태그업 | 포구 위치가 가까워 물리 모델이 자연스럽게 막음. 별도 로직 불필요 |
| 플라이볼 포구 실패(안타) 시 주자 진루 | 이미 #3-b `resolveRunnerAdvances`에서 처리 |
| 번트 팝업 귀루 | 번트 시스템이 Stub 상태 — 번트 구현 시 동일 로직 재사용 |
| 에러에 의한 추가 진루 | → #6 에러 |

## Success Definition

- 외야 플라이아웃 시 3루 주자가 홈 태그업을 시도한다
- 깊은 외야 타구일수록 태그업 성공률이 높아진다
- 얕은 외야 타구(내야 근처)에서는 태그업이 자제된다
- 라인 드라이브 직접 포구 시 주자가 귀루 아웃되는 경우가 발생한다
- 희생플라이(SAC F) 스탯이 정상 기록된다
- 기존 경기 루프(삼진/볼넷/안타/홈런/병살 등) 정상 동작 유지
