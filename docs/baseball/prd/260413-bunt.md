## Goals / Non-Goals

### Goals (Phase 1)
- **G1**: 번트 결정 AI — 상황 기반 attempt 여부 판단
- **G2**: 통합 번트 실행 — 희생번트/기습번트를 `running` 스탯 기반 확률 편향으로 하나의 파이프라인으로 처리 (옵션 A)
- **G3**: 번트 전용 타구 물리 — 3방향 (1루선/P정면/3루선) 단순 모델
- **G4**: 번트 수비 대응 — 가장 가까운 야수 + 선행 주자 우선 송구
- **G5**: 번트 실패 케이스 — 파울, 팝업(IFR 연동), 포스아웃, 번트 안타

- 세이프티 번트 "철회" 메커니즘 (2스트라이크 전 배트 빼기)
- 런다운/태그 플레이 정밀 모델
- 낫아웃(2스트라이크 번트 파울 삼진) — 낫아웃 피처 구현 시 통합

## Design

### 1. 결정 AI — `decideBunt()` 재구현

**입력**: batter, count, runners, outs, inning, scoreDiff (신규 필요)

**출력**: `{ attempt: boolean; intent: 'sacrifice' | 'hit' }`

**판단 로직** (우선순위 순):

---

#### (A) 희생번트 — `intent: 'sacrifice'`

**게이트 조건** (하나라도 불만족 시 즉시 attempt=false)
- 아웃 < 2
- 득점권 주자 있음 (1루 OR 2루)
- 점수차 ≤ 2 (근소하거나 뒤지고 있음)
- 이닝 ≥ 7 (후반) **또는** 1회 초
- 2스트라이크 아님 (파울=삼진 리스크 회피)

**상황 base 확률**
```
base = 0.30
+ 1·2루 또는 만루        → +0.15
+ 점수차 0점(동점) or -1점 → +0.10
+ 9회 초/말 + 근소한 차   → +0.10
```

> **포지션이 아닌 스탯으로**: 투웨이 플레이어(타격하는 투수)를 고려해 `position === 'P'` 같은 하드코딩 분기를 쓰지 않는다. 아래 스탯 모디파이어에서 contact/power가 낮으면 자동으로 multiplier가 2.0 cap에 도달해 투수급 약타자를 그대로 처리한다.

**타자 스탯 모디파이어 (핵심 — 번트 적합성)**

번트는 "일반 타격을 포기하는 선택"이므로 **타자 능력이 높을수록 덜 시도**해야 한다.
`bunt_aptitude` 점수를 계산해 확률에 승산비로 적용:

```
contact_factor = clamp((65 - batter.contact) / 40, -0.5, +0.5)
   // contact 25 → +1.0배, contact 65 → 0배, contact 95 → -0.75배
power_factor   = clamp((60 - batter.power)   / 50, -0.4, +0.4)
   // 강타자일수록 희생번트 금지 (클린업 타자는 절대 번트 X)
eye_factor     = clamp((60 - batter.eye)     / 80, -0.2, +0.2)
   // 선구안 좋은 타자는 볼넷 기대 → 번트 덜 시도
running_factor = clamp((50 - batter.running) / 100, -0.15, +0.1)
   // 느린 주자는 어차피 병살 위험 → 희생번트 유도 (+)
   // 빠른 주자는 차라리 일반 타격 후 내야안타 (-)

stat_multiplier = 1 + contact_factor + power_factor + eye_factor + running_factor
stat_multiplier = clamp(stat_multiplier, 0.1, 2.0)   // 극단값 방지

final_prob = base * stat_multiplier
```

**예시 계산**:
| 타자 유형 | contact | power | eye | running | multiplier | base=0.30 → final |
|---|---|---|---|---|---|---|
| 전통 투수형 약타자 | 25 | 20 | 30 | 40 | **2.0 (cap)** | 0.60 |
| 투웨이 투수 (컨택형) | 55 | 45 | 55 | 60 | 1.42 | 0.43 |
| 투웨이 투수 (강타형) | 70 | 75 | 65 | 60 | 0.75 | 0.23 |
| 평균 타자 | 65 | 60 | 60 | 60 | 1.00 | 0.30 |
| 3~5번 슬러거 | 70 | 85 | 70 | 55 | 0.58 | 0.17 |
| 슈퍼스타 | 85 | 90 | 80 | 75 | 0.15 (near floor) | 0.05 |

→ 전통 투수형은 번트 자주, 투웨이 강타형 투수는 거의 안 함. **포지션 체크 없이** 스탯만으로 자연스럽게 역할 분화.

---

#### (B) 기습번트 — `intent: 'hit'`

**게이트 조건**
- 아웃 < 2
- 주자 **없음** (주자 있으면 희생번트 경로가 우선)
- 2스트라이크 아님

**타자 스탯 모디파이어 (기습번트는 "살기 위한 시도")**

빠르고 정확한 타자일수록 유리. 반대로 느리거나 컨택이 약하면 시도 의미 없음.

```
base = 0.02   // 매우 낮게 시작 (기습번트는 희귀)

running_factor = clamp((batter.running - 60) / 30, 0, 1.0)
   // running 60 → 0, running 90 → +1.0
contact_factor = clamp((batter.contact - 50) / 40, 0, 0.6)
   // 정확한 컨택이 필요 (팝업 방지)
eye_factor     = clamp((60 - batter.eye) / 60, -0.2, +0.2)
   // 선구안 좋은 타자는 공 고르므로 기습번트 덜 시도

// 곱연산 (running이 0이면 시도 안 함)
final_prob = base * (1 + 3 * running_factor) * (1 + 2 * contact_factor) * (1 + eye_factor)
```

**추가 보정**:
- 좌타자 → × 1.3 (드래그 효과)
- 9회 + 동점 + 선두타자 → × 1.5 (결정적 순간)

> ※ 초안에 있었던 "상대 3루수 수비력 < 60 → × 1.2" 는 수비 스탯 읽기가 복잡해져 Phase 1에서 미구현. Phase 2 튜닝 시 재검토.

**예시 계산**:
| 타자 유형 | contact | eye | running | 좌타 | final_prob |
|---|---|---|---|---|---|
| 평균 타자 | 60 | 60 | 55 | X | ~0.02 |
| 빠른 컨택형 | 70 | 55 | 80 | O | ~0.18 |
| 이치로형 | 85 | 60 | 90 | O | ~0.32 |
| 느린 강타자 | 80 | 70 | 45 | X | ~0.02 |

---

#### (C) 우선순위와 배타성

`decideBunt()`는 **순서대로** 체크:
1. 희생번트 게이트 통과 → A 경로 확률 롤
2. (1 실패) 기습번트 게이트 통과 → B 경로 확률 롤
3. (2 실패) `{ attempt: false }`

→ 주자 있으면 A만 체크, 주자 없으면 B만 체크되므로 사실상 배타적.

---

#### (D) 디버그/튜닝 관점

모든 계수는 `BUNT_CONFIG` 상수로 분리:
```ts
export const BUNT_CONFIG = {
  sacrifice_base: 0.30,
  sacrifice_contact_weight: 40,
  sacrifice_power_weight: 50,
  // ...
}
```
캘리브레이션 단계에서 번트 시도/경기 목표(0.3~0.8)에 맞게 계수 조정.

### 2. 번트 실행 파이프라인 — hit-ball.ts 통합

`hitBall()`의 `decideBunt()` 분기에서 호출되는 신규 함수:

```ts
resolveBunt(state, pitch, defenceLineup, intent) → BattingResult
```

**하위 단계**:

**2-1. 컨택 판정**
- 스트라이크 존이 아니면 번트 헛스윙 가능 (볼카운트 증가)
- 컨택 성공률:
  ```
  base = 0.75
  + (batter.contact - 50) / 200        // contact 스탯
  - (pitcher.ball_speed - 80) / 100    // 빠른공 페널티
  - (pitch.ball_break_applied / 200)   // 변화구 페널티
  ```
- 실패 → 헛스윙 (스트라이크 추가)

**2-2. 파울 판정**
- 번트는 파울 비율 높음
- base foul_prob = 0.30
- 2스트라이크면 파울 비율 그대로 적용 → 일반 룰로 파울 처리 (낫아웃 아직 미구현이므로 **2스트라이크 진입 전에 attempt=false로 컷**)

**2-3. 팝업 판정 (번트 실패의 핵심)**
- popup_prob 계산:
  ```
  base = 0.08
  + (pitcher.ball_speed - 80) / 300
  + (contact 실패 근접도)              // 컨택 경계선일수록 팝업 ↑
  - (batter.contact - 50) / 400
  ```
- 팝업 발생 → `ball_type = 'popup'`, 내야 낮은 범위(range ~10~25m)로 `resolveHitResult`와 유사 경로 타되 **기존 IFR 로직 재활용**
  - 주자 1·2루 0~1아웃 → 자동 아웃 (이미 구현됨)

**2-4. 유효 번트 타구 생성 — 종류별 방향성**

번트 종류가 다르면 **노리는 방향 자체가 다르다**. 전통적인 야구 전략에 기반한 방향 선호:

---

**희생번트 — 주자를 어디로 밀 것인가에 따라 방향 결정**

전략 원칙: **선행 주자가 가려는 베이스를 지키는 야수를 끌어내야** 포스/송구가 어려워진다.

| 주자 상황 | 선호 방향 | 이유 |
|---|---|---|
| 1루만 | **1루선** (60%) / P정면 (25%) / 3루선 (15%) | 1B를 타구 잡으러 끌어내면 2루 포스아웃이 어려워짐. 2루 송구는 2B가 커버해야 함 |
| 2루만 | **3루선** (60%) / P정면 (25%) / 1루선 (15%) | 3B를 끌어내면 3루 커버 공백 → 2루 주자가 3루로 쉽게 이동 |
| 1·2루 | **3루선** (55%) / P정면 (25%) / 1루선 (20%) | 3B를 끌어내는 게 최우선 (2루 주자 진루 + 1루 주자도 함께 밀림) |
| 만루 | **3루선** (50%) / P정면 (30%) / 1루선 (20%) | 3B가 홈 송구/3루 포스아웃 판단 어려워짐 |

**타자 스킬 보정 (희생번트)**:
- `contact` ≥ 60 → 선호 방향 확률 +10%p (정확한 타자는 의도대로 굴림)
- `contact` < 40 → 의도 실패 — 방향 분포 균등(1/3씩) + **P정면 확률 +20%p** (약한 번트가 투수 앞으로 가기 쉬움)

---

**기습번트 — 타자가 살기 위해 어디로 보낼 것인가**

전략 원칙: **타자가 1루로 뛰는 시간 동안 야수가 잡아서 송구할 수 없는 곳**으로 보낸다.

| 타자 유형 | 선호 방향 | 이유 |
|---|---|---|
| 좌타 + `running` ≥ 75 | **1루선** (75%) / P정면 (15%) / 3루선 (10%) | 드래그 번트 — 1루 방향으로 몸이 이미 움직이는 중. 1B가 타구 잡으러 나가는 동안 1루 베이스는 비어 있음 |
| 우타 + `running` ≥ 75 | **3루선** (65%) / P정면 (20%) / 1루선 (15%) | 3B에게 가장 먼 포지션 — 송구 거리가 길어 세이프 확률 ↑ |
| 그 외 (느린 타자) | **3루선** (55%) / P정면 (30%) / 1루선 (15%) | 세이프티 번트 기본 — 3루수 반응 늦으면 내야안타 |

**타자 스킬 보정 (기습번트)**:
- `contact` ≥ 70 → 선호 방향 확률 +15%p
- `eye` ≥ 70 → 파울/헛스윙 확률 -5%p (별도 보정, 2-2 단계)

---

**2-5. 타구 물리 파라미터**
- 거리: **5~20m 균일 분포** (방향 선택 후 결정)
  - `contact` ≥ 60 → 경계(15~20m) 편향 — 베이스라인 바로 옆으로 정확히 굴러감
  - `contact` < 40 → 중심(8~12m) 편향 — 투수 앞으로 약하게
- `exit_velocity` = 15~35 km/h (낮은 속도, 거의 굴러감)
- `launch_angle` = -5~5° (거의 땅볼, 팝업 분리 판정은 2-3 단계에서 이미 끝남)
- `theta_h` (수평각) = 방향 선택 결과를 실제 각도로 변환
  - 1루선: +35° ± 5° (우측 파울라인 근처)
  - P정면: 0° ± 5°
  - 3루선: -35° ± 5° (좌측 파울라인 근처)

### 3. 번트 수비 대응 — 신규 함수

`resolveBuntDefense(buntLanding, defenceLineup, runners, outs) → HitResultDetail`

**Step 1: 처리 야수 결정**
- 방향별 후보:
  - `first_base_line` → 1B 또는 P (둘 중 타구에 더 가까운)
  - `pitcher_front` → P (기본), C 보조
  - `third_base_line` → 3B 또는 P

**Step 2: 포구 시간 계산**
- `t_fielding = t_ball_travel + dist / fielder_charge_speed + 0.4`
- `fielder_charge_speed`: 수비수 `running` 기반 6.0~7.5 m/s (돌진 보너스)

**Step 3: 송구 대상 판단 (선행 주자 우선)**

```
if 주자 2루(포스X) 또는 3루 → 홈/3루 송구 불가 (판단 보류, 1루 송구)
if 1루 주자 있음 AND 포스 상태 → 2루 송구 시도
  if (t_fielding + t_throw_to_2B) < t_runner_2B → 2루 포스아웃
  else → 1루 송구로 폴백
else → 1루 송구
```

- `t_runner_2B` = 1루 주자의 리드 + 달리는 시간 (`running` 기반)
- 2루 송구 실패 → **all safe** (번트 안타 + 1루 주자 안전)

**Step 4: 1루 송구 결과**

```
t_to_1B = t_fielding + (dist → 1B) / throw_speed + 0.2
t_batter = 27.43m / batter_run_speed

if t_to_1B < t_batter → 타자 아웃 (희생번트 성공 / 실패)
else → 번트 안타 (single)
```

**Step 5: 실책 처리**
- 번트 처리 에러율 = `ERROR_COEFF * 1.5 * (1 - defence/200)` (번트는 에러 난이도 높음)
- 에러 발생 → `reach_on_error`, 주자 진루

### 4. 결과 매핑 (`at_bat_result`)

| 상황 | result | 기존 재활용 |
|---|---|---|
| 타자 1루 아웃 + 주자 진루 | `'out'` + sac 플래그 | ✅ |
| 타자 1루 세이프 + 주자 진루 (번트 안타) | `'single'` | ✅ |
| 선행 주자 2루 포스아웃 + 타자 1루 세이프 | `'fielders_choice'` | ✅ |
| 2루 포스아웃 + 타자 1루 아웃 (번트 병살) | `'double_play'` | ✅ |
| 팝업 → 야수 포구 | `'out'` + `is_infield_fly` (조건 맞을 때) | ✅ IFR 재활용 |
| 실책 | `'reach_on_error'` | ✅ |
| 파울 | 일반 파울 처리 | ✅ |
| 헛스윙 | 스트라이크 | ✅ |

→ 기존 `advanceRunners`/결과 파이프라인을 **최대한 재활용**. 신규 결과 타입 추가 없음.

### 5. 통계 반영

**BattingResult**에 신규 플래그:
```ts

```

`calc-game-stats.ts` 수정:
- 희생번트 성공 → AB 제외 (타율 계산에서 빠짐)
- SH (sacrifice hits) 카운터 추가

**Non-goal**: SF(희생플라이)와 구분 유지. SF는 이미 처리되거나 별도 이슈.

## 데이터 흐름 요약

```
hitBall()
  ├── decideBunt() → { attempt, intent }
  │   └── 조건 매칭 + 확률 롤
  │
  └── if attempt → resolveBunt()
      ├── 컨택 판정 (실패 → 헛스윙)
      ├── 파울 판정 (실패 → foul)
      ├── 팝업 판정 (실패 → popup → resolveHitResult 경로 + IFR)
      ├── 유효 번트 타구 생성 (방향/거리)
      └── resolveBuntDefense()
          ├── 야수 결정
          ├── 선행 주자 송구 판단
          └── 1루 송구 결과 → out / single / FC / DP / error
```

## Open Questions

### Q1. 번트 대비 수비 위치를 동적으로 바꿀 것인가?

**쉬운 설명**:
실제 야구에선 **"이 타자는 번트 할 것 같다"** 싶으면 수비팀이 1루수·3루수를 평소보다 **3~5걸음 앞으로** 당긴다(코너스 인, corners in). 이 자세에서 번트 타구가 오면 야수가 훨씬 빨리 잡아서 번트 성공률이 뚝 떨어진다. 반대로 강타자가 나오면 번트 걱정 없으니 정상 위치(또는 뒤로).

**질문**: 우리 시뮬레이터가 "이 상황은 번트 예상된다" → 3루수 앞당김 → 번트 어려워짐, 이걸 재현할 것인가?

**선택지**:
- **(A) 안 한다** — 수비 위치는 항상 고정(정상 위치). 구현 단순.
- **(B) 한다** — 번트 확률 높은 상황이면 3루수·1루수를 앞당김. `fielder-positions.ts`에 동적 로직 추가.

**최종 결정**: **(B) — 희생번트 상황에만 한정해서 구현**.
- 사용자 논의 결과, 기습번트는 정의상 수비 허점 찌르기이므로 corners-in 대상이 아님
- 희생번트·스퀴즈 대비 상황에서만 1B·3B 전진
- 기존 시프트 시스템 위에 chain — `applyShift` → `applyCornersIn` 순
- 구현 상세는 아래 "Q1 구현 상세" 참조

### Q2. 번트가 약하게 굴렀을 때 포수(C)도 잡으러 나올 수 있게 할 것인가?

**쉬운 설명**:
번트가 홈플레이트 바로 앞(1~5m)에 약하게 굴러가면, 실제로는 **포수가 마스크 벗고 뛰어나와** 맨손으로 낚아채서 1루로 송구한다. 이 타구는 투수(P)보다 포수가 더 가깝기 때문. 번트 수비에서 포수의 `running`·`throw` 스탯이 중요한 이유가 여기 있음.

**질문**: 번트 타구의 수비 후보에 **C(포수)를 포함**시킬 것인가, 아니면 **P(투수)만** 처리하도록 단순화할 것인가?

**선택지**:
- **(A) P만** — 투수 앞 방향 번트는 전부 P가 처리. 단순.
- **(B) P + C 둘 다** — 낙하 지점 기준 가까운 쪽 선택. 특히 홈 앞 짧은 번트는 C가 처리 → 송구 빠름.

**제안**: **(B)**. 코드 10줄 정도 추가면 되고, 포수 스탯이 번트 수비에 의미를 가져서 선수 차별화가 생김.

### Q3. 스퀴즈 번트(squeeze)를 Phase 1에 넣을 것인가?

**쉬운 설명**:
스퀴즈 번트 = **3루 주자가 홈으로 미리 뛰고, 타자는 반드시 번트를 대야 하는** 특수 작전. 성공하면 공짜 1타점. 실패(헛스윙/파울)하면 달리던 3루 주자가 홈 바로 앞에서 공을 기다리는 포수에게 태그아웃. **All-or-nothing 하이리스크 플레이**.

일반 희생번트와의 차이:
- 희생번트: 1·2루 → 2·3루로 밀기. 3루 주자 있어도 홈으로 뛸지는 수비 반응 보고 판단.
- 스퀴즈: **3루 주자가 먼저 뛴다**. 타자의 번트 실패 = 즉시 아웃.

**질문**: Phase 1에서 스퀴즈를 **별도로 구현**할 것인가, 아니면 3루 주자가 있어도 그냥 일반 희생번트 경로로 처리할 것인가?

**선택지**:
- **(A) 스킵** — 3루 주자 상황도 일반 희생번트로 처리. 번트 성공 후 3루 주자가 홈으로 갈지는 기존 tag-up 로직이 판단(번트 → out 판정 → 태그업 체크).
- **(B) 전용 구현** — `intent: 'squeeze'` 분기, 실패 시 3루 주자 홈 태그아웃 처리.

**제안**: **(A) 스킵**. 스퀴즈는 MLB 실제 빈도도 매우 낮고(~0.05/경기), Phase 1의 득점 개선 목표(+0.2점)에 미치는 영향이 작음. Phase 3에서 고급 전략과 함께 구현.
